## construct

- epic 存放在 `./dev-docs/prd/epic_{epic编号}/epic_{epic编号}_{名称}.md`文件中。
- story 存放在 `./dev-docs/prd/epic_{epic编号}/stories/story_{epic编号}_{story编号}_{epic名称}.md`文件中。
- ui单元测试用例 存放在 `./dev-docs/coding/epic_{编号}/ui_unittest_{epic编号}_{story编号}.md`文件中。

## Goals

- 生成专注于核心业务流程的UI测试文档
- 仅覆盖主要的用户操作路径和关键功能验证
- 确保测试路径简洁明了，便于业务理解和执行
- 输出文档可作为业务验收测试的基准

## Constraints

- 严格禁止包含任何技术细节：代码、元素名称、测试框架等
- 仅关注业务主流程，不包含任何边缘case或异常场景
- 每个测试用例只描述核心的用户交互流程
- 测试路径必须具有业务可执行性
- 使用自然语言描述用户交互流程
- 禁止讨论错误处理、边界条件或异常情况

## Workflow

1. **识别核心业务流程**：从story中提取最主要的用户操作路径
2. **聚焦关键功能点**：仅选择对业务目标至关重要的功能进行测试
3. **设计主流程路径**：为每个核心业务场景设计完整的正向操作路径
4. **验证业务目标**：确保每个测试路径直接验证业务需求的实现

## Output Format:

生成的文档必须为Markdown，结构如下：
```markdown
# [UI测试名称]
> epic编号：[epic编号]
> story编号：[story编号]

## 业务目标
基于故事需求：[Story描述]
本文档专注于验证核心业务流程，确保主要功能按预期工作

## 主流程测试路径

### 主流程 1: [核心业务场景名称]
- **业务价值**：验证[该场景对业务的核心价值]
- **前置条件**：[开始测试前的基本状态]
- **成功路径**：
  1. 用户访问[起始界面]
  2. 用户完成[关键操作1]
  3. 系统展示[预期响应1]
  4. 用户完成[关键操作2]
  5. 系统达到[最终业务状态]
- **验证要点**：[需要确认的核心业务结果]

### 主流程 2: [另一个核心业务场景]
- **业务价值**：验证[该场景对业务的核心价值]
- **前置条件**：[开始测试前的基本状态]
- **成功路径**：
  1. [步骤描述]
  2. [步骤描述]
  3. [步骤描述]
- **验证要点**：[需要确认的核心业务结果]
```
文档保存在项目根目录的`./dev-docs/coding/epic_{编号}/ui_unittest_{epic编号}_{story编号}.md`文件。
